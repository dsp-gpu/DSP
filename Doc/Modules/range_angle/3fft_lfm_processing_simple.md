## 🎯 Что такое ЛЧМ-сигнал простыми словами

Представь сирену полицейской машины — она меняет частоту. **ЛЧМ (Линейно-Частотно-Модулированный)** сигнал делает то же самое: **частота плавно растёт** (или падает) во времени.

```
Частота:  ████████████████▲
          │          ╱
          │        ╱
          │      ╱
          │    ╱
          │  ╱
          └──────────────▶ Время
```

Например: сигнал начинается на 1 МГц и заканчивается на 6 МГц за 10 микросекунд.

**Зачем такой странный сигнал?** Потому что это очень хитрый способ "упаковать" информацию, а потом её "распаковать" с огромным выигрышем.

---

## 🤔 В чём проблема и зачем нужно 2FFT?

### Без сжатия: есть выбор между двух зол

**Хочешь точно измерить расстояние до цели?**
- Нужен **короткий импульс** → хорошее разрешение
- Но короткий импульс → **слабый сигнал** → не улетит далеко

**Хочешь засечь цель на большом расстоянии?**
- Нужен **длинный и мощный импульс** → хорошая дальность
- Но длинный импульс → **плохое разрешение** → не видно деталей

😱 Кажется, невозможно сочетать оба желания!

### ЛЧМ + 2FFT = решение этой дилеммы

ЛЧМ позволяет **послать длинный мощный сигнал**, а потом математически **сжать его в короткий** — как будто он и был коротким изначально!

- Длинный импульс → далеко летит ✅
- После сжатия — как короткий → точное расстояние ✅

Выигрыш называют **коэффициентом сжатия D = B·T**. Например, полоса B=5 МГц, длительность T=10 мкс → D = 50 раз!

## 🔧 Что такое FFT и почему нужно ДВА?

### FFT (Быстрое Преобразование Фурье) за 30 секунд

FFT — это математическая машина, которая берёт сигнал **во времени** и показывает, **из каких частот** он состоит.

```
FFT:
Сигнал(время) ──────→ Спектр(частоты)
  🌊🌊🌊🌊        →    📊 (столбики на разных частотах)
```

Например, аккорд на гитаре → FFT покажет три ноты, из которых он состоит.

### Почему ДВА FFT?

Нельзя обработать ЛЧМ одним FFT, потому что ЛЧМ-сигнал "размазан" по всем частотам — его спектр широкий и плоский, никакого пика нет.

Нужно сначала **подготовить** сигнал, а потом **применить FFT**.
Это "подготовка + FFT" и есть то, что называют **двойным FFT** или **2FFT**.

## 🛠️ Три способа использовать 2FFT

### Способ 1: Согласованный фильтр (самый классический)

**Аналогия**: У тебя есть ключ (опорный сигнал). Ты проверяешь — подходит ли твой ключ к замку (принятому сигналу). Если подходит — вот тут и сидит цель!

```
Шаг 1: FFT принятого сигнала
Шаг 2: Умножить на "перевёрнутый" спектр опорного
Шаг 3: IFFT — получаем сжатый импульс с пиком
```

**Почему 2 FFT лучше чем перебор?**

| | Прямой перебор | 2 FFT |
|-|---------------|-------|
| Операций для 1024 точек | ~1 000 000 | ~10 000 |
| **Быстрее в** | 1× | **~100×** |

Для GPU это критично: **можно обработать в 100 раз больше данных** за то же время!

---

### Способ 2: Dechirp + FFT (Stretch Processing) — самый умный

**Аналогия**: Настрой радиоприёмник точно на нужную волну — и вместо шума услышишь чистую музыку.

**Идея**: Если умножить принятый ЛЧМ на "зеркальную копию" опорного ЛЧМ — все прыгающие частоты **скомпенсируются** и сигнал превратится в обычную синусоиду. А частота этой синусоиды говорит, **на каком расстоянии** находится цель!

```
ЛЧМ принятый (частота скачет вверх):    ╱╱╱╱╱╱╱╱╱╱
×
Опорный (частота скачет вниз):          ╲╲╲╲╲╲╲╲╲╲
                                      = ──────────  (чистый тон!)
```

```
Шаг 1: Умножить принятый на зеркальный опорный (dechirp)
Шаг 2: FFT — получаем пик, по частоте которого считаем расстояние
```

**Главное преимущество**: сигнал после умножения — **узкополосный**!
- Обычный ЛЧМ занимает 5 МГц полосы → нужен дорогой быстрый АЦП
- После dechirp остаётся только 0.1 МГц → нужен простой дешёвый АЦП
- Экономия в **10–1000 раз**! 💰

**Связь с GPUWorkLib**: наш модуль `HeterodyneDechirp` уже делает Шаг 1! Добавить FFTProcessor = получим полный Stretch Processing.

### Способ 3: Range-Azimuth-Elevation куб (3D FFT) — наш случай! 🎲

> 🎯 **Это именно наша задача**: 16×16 антенн (URA) × 1 300 000 точек

**Аналогия**: Представь **сетку 16×16 фотографов** — как стадион с камерами. Каждая камера смотрит в одну точку неба, но под чуть другим углом. Горизонтальный ряд даёт **азимут** (лево-право), вертикальный — **элевацию** (вверх-вниз). Сложи все снимки — получишь **3D карту неба: откуда, как высоко и как далеко** находится объект.

---

#### Структура нашей 3D матрицы данных

```
16×16 антенная решётка (вид сверху):

     Азимут →  az=0  az=1  az=2  ...  az=15
Элев ↓
el=0          [A]   [A]   [A]         [A]
el=1          [A]   [A]   [A]         [A]
el=2          [A]   [A]   [A]         [A]
...
el=15         [A]   [A]   [A]         [A]

Каждая антенна [A] пишет 1 300 000 точек (83 нс каждая).
В памяти: плоский массив [256 × 1 300 000], где 256 = 16×16.
```

---

#### Шаг 1 — Range FFT: каждая антенна → дальностный профиль

```python
# Для КАЖДОЙ из 256 антенн (все параллельно на GPU!):
dechirped[a] = data[a] * conj(ref_lfm)    # убираем ЛЧМ-модуляцию
range_fft[a] = FFT(dechirped[a])          # получаем профиль дальностей

# fs = 12 МГц, B = 10 МГц (девиация -5→+5 МГц), N = 1 300 000
# Разрешение по дальности: ΔR = c / (2B) = 3×10⁸ / (2 × 10⁶) = 15 м  ← точнее Capon!
# Несущая: f_c = (430+440)/2 = 435 МГц, λ = 0.69 м
# Полезных бинов дальности: 650 000 бинов  (от 0 до 9 750 км)
```

После Шага 1 матрица стала `[256 × 650 000]`:

```
              ← 650 000 дальностных бинов (шаг 15м) →

антенна  0,0  [R=0  R=15м  R=30м  ...  R=9750км]
антенна  0,1  [R=0  R=15м  R=30м  ...  R=9750км]
  ...
антенна 15,15 [R=0  R=15м  R=30м  ...  R=9750км]
```

Каждая антенна видит одни и те же дальности, но **с разной фазой** (2D сдвиг!).

---

#### Шаг 2 — 2D Beam FFT: каждая дальность → азимут И элевация

**Возьмём один дальностный срез (например, R = 150 км, бин №10 000):**

```
Это срез нашей 3D матрицы — таблица 16×16 комплексных чисел:

          az=0      az=1      az=2   ...   az=15
el=0   [0.7+0.5j] [0.3+0.8j] ...         [...]    ← фаза растёт →  (по горизонтали = азимут)
el=1   [0.5+0.7j] [0.1+0.9j] ...         [...]
el=2   [-0.4+0.6j] ...                   [...]
  ↓                                               ← фаза растёт ↓  (по вертикали = элевация)
el=15  [0.2-0.7j] ...                    [...]
```

По горизонтали (az=0..15) фаза нарастает с шагом `Δφ_az = 2π×d×sin(θ_az)/λ` → это **азимут**.
По вертикали (el=0..15) фаза нарастает с шагом `Δφ_el = 2π×d×sin(θ_el)/λ` → это **элевация**.

**2D FFT от этой таблицы 16×16 находит ОБА шага сразу → (θ_az, θ_el) за один вызов!** 🚀

```python
# Для КАЖДОГО из 650 000 дальностных бинов (все параллельно!):
beam_2d[:, :, range_bin] = FFT2D(range_fft_2d[:, :, range_bin])
#                                 ↑ это таблица 16×16 от 256 антенн

# Пик на позиции (K_az, K_el) → θ_az = arcsin(K_az × λ / (16×d))
#                               → θ_el = arcsin(K_el × λ / (16×d))
```

Да, мы берём 650 000 раз **2D FFT от матрицы 16×16**. На GPU **все 650 000 FFT одновременно** — одна команда hipFFT с `batch=650000`, размер `[16×16]`. 🚀

---

#### Итог трёх шагов — 3D куб!

```
Вход:     [16 × 16 × 1 300 000]  = [256 антенн × 1.3М точек]
            ↓ Dechirp + Range FFT (batch=256, длина=1.3М)
            ↓ Crop до 650 000 полезных бинов
Промежут: [256 × 650 000]
            ↓ Transpose → [650K × 256] = [650K × 16 × 16]
            ↓ 2D Beam FFT (batch=650K, размер=[16×16])
            ↓ 2D fftshift по осям азимута и элевации
Выход:    [650 000 × 16 × 16]  ← 3D КУUБ мощности!

Размерность куба:
  Ось 0: 650 000 дальностных срезов (шаг 15 м, от 0 до 9 750 км)
  Ось 1:      16 азимутальных бинов (шаг ~7.2°, от -53° до +53°)
  Ось 2:      16 бинов элевации     (шаг ~7.2°, от -53° до +53°)
```

## 🥊 Почему 2D FFT лучше Capon (текущий подход)

### Как работает наш Capon прямо сейчас

Смотрим код `CaponProcessor`:

```
Шаг 1: R = (1/N) × Y × Y^H   ← ковариационная матрица [256×256]
Шаг 2: R^{-1}                 ← обращение через Cholesky
Шаг 3: W = R^{-1} × U         ← весовые векторы для каждого направления
Шаг 4: Y_beam = W^H × Y       ← адаптивное диаграммообразование
Шаг 5: FFT(Y_beam[m,:])        ← отдельно FFT для каждого луча
```

### Считаем операции на наших числах

**P = 256 антенн, N = 1 300 000 отсчётов, M = 256 направлений:**

```
CAPON (текущий подход):

  Шаг 1 — ковариация R = Y×Y^H:
    256 × 256 × 1 300 000 = 85 200 000 000 операций  (85 млрд!)

  Шаг 4 — беамформинг Y_beam = W^H × Y:
    256 × 256 × 1 300 000 = 85 200 000 000 операций  (85 млрд!)

  Шаг 5 — FFT каждого луча:
    256 × 1 300 000 × 20  =  6 656 000 000 операций

  ИТОГО: ~177 000 000 000 операций  (177 млрд)
  ════════════════════════════════════════════════

3D FFT (новый подход):

  Range FFT (dechirp + FFT, batch=256):
    256 × 1 300 000 × 20  =  6 656 000 000 операций

  2D Beam FFT [16×16], batch=650 000:
    650 000 × 16 × 16 × 8  =  1 331 200 000 операций  (≈ 1.3 млрд)

  ИТОГО: ~8 000 000 000 операций  (8 млрд)
  ════════════════════════════════════════════════
```

```
┌─────────────────────────────────────────────┐
│  Capon:   177 млрд операций  ████████████  │
│  3D FFT:    8 млрд операций  █             │
│                                             │
│  3D FFT БЫСТРЕЕ В  22 РАЗА  🚀             │
└─────────────────────────────────────────────┘
```

### Но это не единственное преимущество!

**Проблема Capon: ковариационная матрица усредняет ВСЕ дальности сразу**

```
Capon считает одну матрицу R от всех 1.3М точек:

  R = среднее по ВСЕМУ времени = не знает где цель по дальности

  Цель на 100 км  ─┐
  Цель на 5000 км ─┤─→ всё смешивается в одной матрице R
  Шум на 200 км   ─┘

  Результат: веса W рассчитаны для "средней" картины
  → углы получаются неточные, если цели на разных дальностях!
```

**2D FFT решает это автоматически:**

```
2D FFT сначала разделяет по дальности, ПОТОМ считает угол:

  Бин 100 км:  беамформинг только для этой дальности ✅
  Бин 5000 км: беамформинг только для этой дальности ✅
  Бин 200 км:  беамформинг только для этой дальности ✅

  Каждая дальность обрабатывается независимо!
  Цели на разных дальностях не мешают друг другу!
```

| Выигрыш SNR | 60.3 дБ | **60.3 дБ** (одинаково!) |
| Код готов в GPUWorkLib | 100% | **~85% уже есть!** |

### Когда Capon лучше?

Capon выигрывает в одном: **угловое сверхразрешение** (два источника ближе чем λ/N·d). Но это нужно не всегда, и Capon не знает на каком расстоянии цель!

### Профессиональный подход: сначала 3D FFT, потом Capon точечно

```
Этап 1: 3D FFT (быстро, 8 млрд операций)
         → получаем куб всей сцены: 16×16 углов × 650 000 дальностей
         → находим интересные бины (где есть цели)

Этап 2: Capon только на выбранных дальностных срезах (например, 5 срезов)
         → 5 × 256² × ~500 точек = 164 млн операций
         → получаем сверхточное угловое разрешение только там где нужно

Итого: быстро + точно там где важно = лучшее из двух миров 🏆
```

## 🎯 Что делать с Range-Angle картой: извлечение угла и дальности

### Откуда берётся угол из бина Beam FFT

Физика: волна под углом θ приходит на каждую антенну с **одним и тем же сдвигом фазы** Δφ. Beam FFT находит этот сдвиг и сразу даёт угол:

```
Δφ = 2π × d × sin(θ) / λ

Бин K (после fftshift)  →  sin(θ) = K × λ / (N × d)  →  θ = arcsin(...)
```

где `d` — расстояние между антеннами, `λ = c / f_несущей`.

**Нумерация бинов после fftshift:**

```
  Бин -128 → θ ≈ -90°  (крайний левый)
  Бин    0 → θ =   0°  (прямо вперёд, нормаль)
  Бин  +127 → θ ≈ +90°  (крайний правый)

  При d = λ/2 → весь диапазон ±90° без ложных лепестков
```

Угловое разрешение с 256 антеннами при `d = λ/2`:
```
Δθ ≈ 2 / 256 = 0.0078 рад ≈ 0.45°  (у нормали, θ=0°)
С окном Хэмминга: ~0.66°  (×1.46 уширение)
```

### Полная цепочка: карта → цель

```python
import numpy as np

# Параметры антенной решётки (16×16 URA, UHF 435 МГц)
fc     = 435e6        # несущая частота Гц
lam    = 3e8 / fc     # длина волны = 0.690 м
d      = lam / 2      # шаг λ/2 = 0.345 м ≈ 35 см
N_az   = 16           # антенн по азимуту
N_el   = 16           # антенн по элевации

# Параметры сигнала
fs, B, N = 12e6, 10e6, 1_300_000
T  = N / fs                    # 108.3 мс
mu = B / T                     # chirp rate = 92.3 МГц/с

# --- 3D куб мощности [N_range × N_az × N_el] ---
cube_3d = np.abs(beam_3d_fft) ** 2    # [650000 × 16 × 16]

# --- Найти 3D пик ---
r_bin, az_bin, el_bin = np.unravel_index(np.argmax(cube_3d), cube_3d.shape)

# --- Бин → азимут и элевация ---
# После fftshift: центр = бин 8 (из 0..15)
k_az  = az_bin - N_az // 2    # центрированный (-7..+8)
k_el  = el_bin - N_el // 2    # центрированный (-7..+8)

# sin(θ) = k × λ / (N_side × d) = k × 2 / N_side  (т.к. d = λ/2)
theta_az = np.degrees(np.arcsin(np.clip(k_az * 2.0 / N_az, -1, 1)))
theta_el = np.degrees(np.arcsin(np.clip(k_el * 2.0 / N_el, -1, 1)))

# --- Бин → дальность ---
f_beat = r_bin * fs / N        # частота биения
R      = f_beat * 3e8 / (2 * mu)

print(f"Цель: R = {R/1000:.1f} км,  θ_az = {theta_az:.1f}°,  θ_el = {theta_el:.1f}°")
# Пример: Цель: R = 150.2 км,  θ_az = 7.2°,  θ_el = -14.4°
```

### Sub-bin интерполяция: точность без памяти

Пик может попасть между бинами → параболическая интерполяция по 3 точкам даёт точность в **5–10 раз лучше** без увеличения памяти:

```python
def parabolic_interp(mag, k):
    """Уточнить позицию пика до долей бина."""
    a, b, g = mag[k-1], mag[k], mag[k+1]
    delta = 0.5 * (a - g) / (a - 2*b + g)
    return k + delta   # дробный индекс

# Уточняем три индекса (дальность + два угла):
r_fine  = parabolic_interp(cube_3d[:, az_bin, el_bin], r_bin)
az_fine = parabolic_interp(cube_3d[r_bin, :, el_bin],  az_bin)
el_fine = parabolic_interp(cube_3d[r_bin, az_bin, :],  el_bin)

k_az_fine = az_fine - N_az // 2
k_el_fine = el_fine - N_el // 2
theta_az  = np.degrees(np.arcsin(np.clip(k_az_fine * 2.0 / N_az, -1, 1)))
theta_el  = np.degrees(np.arcsin(np.clip(k_el_fine * 2.0 / N_el, -1, 1)))
R         = (r_fine * fs / N) * 3e8 / (2 * mu)
```

## 🪟 Окно Хэмминга: где применять и зачем

### Правило: окно применяется ПОСЛЕ дечирпа, ПЕРЕД FFT — к реальным данным

```
✅ Правильный порядок:
   сигнал → дечирп → × окно → FFT

❌ Неправильно:
   сигнал → × окно → дечирп → FFT   (окно искажает дечирп)
```

**Два отдельных окна для двух FFT:**

```python
# Range window — по 1 300 000 отсчётам (после дечирпа)
range_win  = np.hamming(N)                             # длина 1 300 000
dechirped  = data * np.conj(ref_lfm)                  # дечирп
dechirped_w = dechirped * range_win                   # окно
range_fft  = np.fft.fft(dechirped_w, n=nfft, axis=1) # Range FFT

# Beam window — по 256 антеннам (перед Beam FFT)
beam_win   = np.hamming(N_ant)                         # длина 256
range_fft_w = range_fft * beam_win[:, np.newaxis]     # окно по антеннам
beam_fft   = np.fft.fftshift(
    np.fft.fft(range_fft_w, axis=0), axes=0)          # Beam FFT
```

| Окно | Боковые лепестки | Потеря разрешения | Выбрать если... |
|------|-----------------|-------------------|-----------------|
| Нет | -13 дБ | 0% | одна цель, нет помех |
| Хэнн | -32 дБ | +50% | общий случай |
| **Хэмминг** | **-43 дБ** | **+46%** | **наш выбор** |
| Блэкман | -58 дБ | +73% | слабая цель рядом с сильной |
| Taylor -40дБ | -40 дБ | +25% | радар (баланс) |

### 🎯 Боковые лепестки — объяснение для чайника

#### Представь прожектор

Когда направляешь антенную решётку в сторону цели — это как направить прожектор. Идеальный прожектор светит **строго в одну точку**. Но реальный всегда даёт немного света в стороны — это и есть **боковые лепестки**.

```
Идеальная антенна:        Реальная антенна (без окна):

        ▲  главный луч              ▲
        █                        ░  █  ░
        █                      ░ █  █  █ ░
        █                     ░  █  █  █  ░
   ─────█─────  0°         ───░──█──█──█──░─── 0°
                              ░  (-13 дБ)  ░
```

#### Что происходит без окна (-13 дБ)

Представь: летят два самолёта рядом по углу.
- **Самолёт А** — бомбардировщик, большой, мощный отражатель
- **Самолёт Б** — маленький истребитель, в 20 раз слабее

Когда радар смотрит **на самолёт Б** — главный луч на нём. Но боковой лепесток от **самолёта А** заливает туда же −13 дБ шума. Поскольку А сам по себе мощный — его лепесток **перекрывает** слабый Б.

**Результат**: вместо самолёта Б радар видит «эхо» от самолёта А. Это ложная цель 👻

#### Что даёт окно Хэмминга (-43 дБ)

```
Без окна:    боковой лепесток = -13 дБ  →  в 20 раз слабее главного
С Хэммингом: боковой лепесток = -43 дБ  →  в 20 000 раз слабее главного

Разница: 30 дБ = в 1000 раз лучше подавление!
```

```
С Хэммингом:

           ▲  главный луч
           █
           █
    ───────█───────  0°
      ▏         ▏   ← боковые лепестки теперь крошечные
   (-43 дБ)  (-43 дБ)
```

#### Аналогия 🔊

Представь концертный зал. **Главный луч** — это прямой звук со сцены. **Боковые лепестки** — это эхо от стен.

| | Без окна | С Хэммингом |
|---|---|---|
| Эхо со стен | Громкое, −13 дБ | Тихое, −43 дБ |
| Слышишь тихий шёпот? | Нет — эхо заглушает | Да — эхо почти исчезло |

#### Цена: небольшое уширение луча

```
Без окна:    ширина луча ≈ 3.6°  (острый, точный)
С Хэммингом: ширина луча ≈ 5.3°  (чуть шире, +46%)
```

Два самолёта разделённых **меньше 3°** без окна видны раздельно, с окном сливаются. Но это редкая ситуация. Зато подавление помех выросло в 1000 раз — почти всегда выгодный обмен.

> **Итог одной строчкой**: Окно Хэмминга — это как **шумоподавление в наушниках**: чуть теряешь чёткость, зато не слышишь шум самолёта. 🎧

#### Реализация в range_angle (BeamFftOp)

Для 2D антенной решётки применяется **сепарабельное** 2D окно — произведение двух 1D Хэммингов:

```cpp
// w2d[az, el] = hamming(az, N_az) × hamming(el, N_el)
// Генерируется на CPU (всего 16×16 = 256 float), заливается на GPU один раз в InitPlan()
// Применяется kernel'ом apply_beam_window_kernel ПЕРЕД hipfftExecC2C
```

```python
# Python-эквивалент:
beam_win = np.outer(np.hamming(N_az), np.hamming(N_el))  # [16×16]
data_windowed = range_3d * beam_win[:, :, np.newaxis]    # умножить перед Beam FFT
```

**Порядок в BeamFftOp::Execute():**
```
1. apply_beam_window_kernel  ← умножить [n_range_bins × 16 × 16] на окно [16×16]
2. hipfftExecC2C             ← 2D FFT
3. fftshift2d_kernel         ← центрировать нулевой угол
```

## 📐 Zero-padding: размер 2^n и не только

### Главное правило: разрешение ≠ zero-padding

```
Zero-padding = интерполяция спектра (больше точек на графике)
             ≠ улучшение разрешения (зависит только от длины сигнала T)

Аналогия: zoom-in фото ≠ более резкое фото
```

### Когда нужен для Range FFT (N = 1 300 000)

```
N = 1 300 000

Следующая степень 2:  2^21 = 2 097 152  ← рекомендуется (hipFFT максимально быстр)
2× интерполяция:      2^22 = 4 194 304  ← только если нужна sub-bin точность

Память (256 антенн × complex64):
  Без pad:   256 × 1.3М × 8 =  2.66 ГБ
  2^21:      256 × 2.1М × 8 =  4.29 ГБ  ← выбор
  2^22:      256 × 4.2М × 8 =  8.59 ГБ  ← много
```

### Если число антенн НЕ равно 2^n — добавить нули!

256 антенн = 2^8 — всё хорошо. Но если было бы, например, 200 антенн:

```python
N_ant_real = 200       # реальное число антенн
N_ant_fft  = 256       # ближайшая степень 2  (2^8)
# или N_ant_fft = 512  # следующая степень 2  (2^9) — лучше угловая интерполяция

# Zero-pad по оси антенн:
# data: [200 × N_range]
data_padded = np.zeros((N_ant_fft, N_range), dtype=complex)
data_padded[:N_ant_real, :] = data   # реальные антенны вверху, нули внизу
```

**Важно: где ставить нули?**

```
✅ Нули ПОСЛЕ реальных антенн (в конце массива):
   [A0, A1, ..., A199, 0, 0, ..., 0]   ← правильно

❌ Нули В СЕРЕДИНЕ или перемешанные — нельзя, сломает пространственную когерентность
```

**Что даёт pad антенн до 2^n:**
- Ускорение hipFFT (степень 2 → оптимальные алгоритмы)
- Более мелкая угловая сетка (интерполяция), но разрешение не меняется
- Разрешение по-прежнему = `λ / (N_real × d)` — от реальных антенн, не от нулей!

### Порядок операций (полный пайплайн с pad, 3D куб)

```python
N      = 1_300_000
nfft_r = 1 << (N - 1).bit_length()   # 2^21 = 2 097 152  (Range FFT)
N_az   = 16                            # антенн по азимуту (уже 2^4)
N_el   = 16                            # антенн по элевации (уже 2^4)

range_win = np.hamming(N)
beam_win  = np.outer(np.hamming(N_az), np.hamming(N_el))  # [16×16] 2D окно

# data: [256 × 1 300 000]  (256 = 16_az × 16_el)

# 1. Дечирп + Range FFT → crop
dechirped = data * np.conj(ref_lfm)                     # [256 × 1.3M]
range_fft = np.fft.fft(
    dechirped * range_win, n=nfft_r, axis=1)            # [256 × 2 097 152]
range_fft = range_fft[:, :650_000]                      # [256 × 650 000]

# 2. Reshape → 3D (только изменение формы, данные не копируются!)
range_3d  = range_fft.reshape(N_az, N_el, 650_000)     # [16 × 16 × 650 000]

# 3. 2D Beam window → 2D FFT по осям 0 и 1
range_3d_w = range_3d * beam_win[:, :, np.newaxis]     # 2D окно Хэмминга
beam_3d    = np.fft.fft2(range_3d_w, axes=(0, 1))      # [16 × 16 × 650 000]

# 4. 2D fftshift по обеим угловым осям
beam_3d    = np.fft.fftshift(beam_3d, axes=(0, 1))

# 5. Куб мощности + поиск пика
cube_3d   = np.abs(beam_3d) ** 2                        # [16 × 16 × 650 000]
az_bin, el_bin, r_bin = np.unravel_index(
    np.argmax(cube_3d), cube_3d.shape)

# 6. Бины → физические координаты (см. раздел выше)
```

## 📝 Краткий итог

| Метод | Что даёт | Когда использовать |
|-------|---------|-------------------|
| **Согласованный фильтр** (FFT + IFFT) | Расстояние, максимальный SNR | Универсальная задача |
| **Stretch Processing** (dechirp + FFT) | Расстояние + дешёвый АЦП | ШПС-радар, SAR |
| **Range-Angle 2D FFT** | Расстояние + угол | Антенная решётка, наш случай |
| **FrFT** | Оценка параметров LFM | Слабые сигналы, неизвестная частота |

**Главный смысл 3D FFT**: вместо дорогих O(N²) вычислений Capon получаем O(N log N) — **в ~22 раза быстрее** — с полной 3D картиной: дальность + азимут + элевация за один проход. Выход — куб `[650K × 16 × 16]`, 0.33 ГБ, на GPU за считанные миллисекунды. 🚀
